WorkFlow Hub - 정책서
목적
본 문서는 WorkFlow Hub 서비스의 운영 정책을 정의한다.
모든 사용자, 기획, 디자인, 개발 및 AI는 본 정책을 기준으로 기능을 설계하고 구현한다.

1. 권한 정책
권한 레벨
	•	Owner
	•	Manager
	•	Member
권한 우선순위
Owner > Manager > Member

계정 관리
계정 생성 및 권한 부여는 Owner만 가능하다.

권한 변경
권한 변경은 즉시 적용된다.
모든 권한 변경 이력은 저장한다.

2. 업무 요청 정책
요청 생성
업무 요청은 Manager 또는 Owner만 생성할 수 있다.

필수 입력값
	•	업무명
	•	업무 상세내역
	•	시작일
	•	마감일
	•	전달 팀
	•	중요도

요청 알림
업무 요청 생성 시 수신 팀의 업무요청 영역에 즉시 표시된다.

수신 및 배정
업무요청은 먼저 전달 팀으로 수신된다.
전달 팀의 Manager가 담당자를 배정한다.
담당자 배정 전 업무요청은 미배정 업무로 표시한다.
담당자 배정 후 담당자가 수락 또는 거절한다.

수락
업무 요청은 수락 또는 거절 처리가 필수이다.
수락 시 업무는 홈 화면의 업무요청 영역에 표시된다.
업무는 자동 생성되지 않는다.
사용자는 원하는 날짜를 선택하여 Weekly Tasks에 추가할 수 있다.

거절
업무 요청 거절 시 거절 사유 입력은 필수이다.
MVP에서는 고정 거절 사유 목록을 사용한다.
거절 사유 목록
	•	일정 불가
	•	담당 범위 아님
	•	리소스 부족
	•	요청 내용 불명확
	•	기타
기타 선택 시 상세 사유 입력은 필수이다.
Owner가 거절 사유를 직접 관리하는 기능은 추후 확장으로 둔다.

상태 흐름
	•	신규 요청
	•	미배정 업무
	•	수락 대기
	•	수락
	•	거절

업무요청 수정
업무 요청은 수락 후에도 수정할 수 있다.
수정 가능 항목
	•	업무명
	•	업무 상세내용
	•	시작일
	•	마감일
	•	중요도
수정 알림
수정 발생 시 담당자에게 알림을 발송한다.
반영 정책
업무 요청 수정 시 기존 업무항목은 자동 변경되지 않는다.
담당자는 수정 알림을 확인한 후 직접 업무항목에 반영할 수 있다.
수정 이력
모든 수정 이력은 데이터베이스에 저장한다.

업무요청 미응답 정책
24시간 미응답
담당자에게 1차 재알림 발송

48시간 미응답
담당자에게 2차 재알림 발송
요청자에게 미응답 상태 알림 발송

이후
수락 대기 상태 유지
자동 거절하지 않는다.

거절 이력
거절 이력은 최근 1주일간 보관한다.
조회 범위
Member
	•	본인 이력
Manager
	•	팀 전체 이력
Owner
	•	전체 이력

3. 업무항목 정책
업무항목 생성
업무항목은 다음 두 가지 방식으로 생성할 수 있다.
직접 생성
Weekly Tasks에서 직접 생성
업무요청 기반 생성
업무요청 수락 후 원하는 날짜를 선택하여 Weekly Tasks에 추가

필수 입력값
	•	업무명
	•	시작일
	•	마감일
	•	업무유형

업무유형
유형
	•	일반
	•	고정
	•	긴급
기본값
일반

참여자 정책
참여자는 사용자가 직접 입력하지 않는다.
직접 생성
생성자를 참여자로 자동 등록한다.
업무요청 생성
업무 요청을 통해 생성된 업무는 담당자를 참여자로 자동 등록한다.
참여자 수정
Owner, Manager만 가능

완료 정책
업무항목에는 완료 상태가 존재하지 않는다.
작업세션만 완료 처리할 수 있다.

진행중 표시
진행중은 별도의 상태값을 사용하지 않는다.
현재 날짜가 업무 기간에 포함되는 경우 해당 날짜의 업무 카드를 강조 표시한다.

지연중 정책
다음 조건을 모두 만족하면 지연중으로 판단한다.
	•	마감일 경과
	•	미완료 작업세션 존재

지연중 표시
지연중 태그를 표시한다.

자동 숨김
다음 조건을 모두 만족하면 Daily Todo에서 자동 숨김 처리한다.
	•	마감일 경과
	•	미완료 작업세션 없음

4. Weekly Tasks 정책
목적
주간 업무를 계획하고 관리하는 영역이다.

작성 규칙
매주 월요일 일주일치 업무를 작성한다.
주간회의를 통해 Weekly Tasks를 확정한다.

추가 업무
다음 사유로 추가 업무 등록이 가능하다.
	•	업무 요청
	•	회의 결과
	•	긴급 업무
	•	기타 업무

자동 배치
업무항목 생성 시 시작일 ~ 마감일 기간에 해당하는 날짜의 Weekly Tasks에 자동 표시한다.

5. Daily Todo 정책
목적
선택한 업무항목의 작업세션을 관리하는 영역이다.

업무 선택
사용자는 Weekly Tasks에서 업무항목을 선택한다.
선택된 업무항목의 작업세션 작성 영역이 열린다.

표시 정책
업무항목명은 반복 표시하지 않는다.
작업세션만 표시한다.

6. 작업세션 정책
생성
작업세션은 업무항목 내부에서 생성한다.

필수 입력값
	•	업무카테고리
	•	세부업무명

최초 생성
최초 생성 시
	•	업무카테고리
	•	세부업무명
만 입력한다.

시간 입력
시간은 최초 생성 시 입력하지 않는다.
시간 영역을 더블클릭하여 입력한다.

입력값
	•	시작시간
	•	종료시간

소요시간
시작시간과 종료시간을 기준으로 자동 계산한다.

시간 정책
모든 시간은 수동 입력한다.
타이머 기능은 제공하지 않는다.

완료 처리
작업세션만 완료 체크가 가능하다.

삭제 처리
작업세션은 물리 삭제하지 않는다.
삭제 시 deleted_at, deleted_by를 기록하여 소프트 삭제한다.
소프트 삭제된 작업세션은 일반 사용자 화면에서 숨김 처리한다.
업무시간 및 생산성 분석에서는 기본 제외한다.
감사 및 이력 확인을 위해 데이터베이스에는 보관한다.

수정 가능 항목
	•	업무카테고리
	•	세부업무명
	•	시작시간
	•	종료시간

수정 이력
사용자 화면에는 표시하지 않는다.
모든 수정 이력은 데이터베이스에 저장한다.

7. 업무카테고리 정책
관리 권한
업무카테고리는 Owner만 관리할 수 있다.

가능 기능
	•	생성
	•	수정
	•	비활성화

삭제
삭제는 불가능하다.

비활성화
비활성화된 카테고리는 신규 작업세션에서 선택할 수 없다.

적용 시점
카테고리 변경 사항은 매월 1일부터 적용한다.
생성, 수정, 비활성화 요청은 즉시 사용 상태에 반영하지 않고 예약 변경으로 저장한다.
예약 변경에는 effective_from 값을 저장한다.
effective_from은 다음 월 1일로 설정한다.
매월 1일에 예약 변경을 실제 카테고리 상태에 반영한다.
기존 작업세션 데이터의 카테고리 정보는 변경하지 않는다.

8. 연차 정책
연차 유형
	•	종일 연차
	•	오전 반차
	•	오후 반차

신청
신청 사유 입력은 필수이다.

승인
Member 신청 → Manager 승인
Manager 신청 → Owner 승인

반려
반려 사유 입력은 필수이다.

취소
연차 시작 전까지만 취소 가능하다.

표시 정책
연차 상세 정보는 연차 관리 탭에서만 조회할 수 있다.
캘린더에는 연차 정보를 표시하지 않는다.

홈 화면 표시
연차 사용 중인 경우 업무 위젯은 비활성화 상태로 표시한다.
팀원 업무 현황에서는 연차 상태만 표시한다.

9. 회의 정책
회의 생성
회의는 캘린더에 등록한다.

회의록 저장
	•	회의 제목
	•	회의 참여자
	•	회의 내용
저장

AI 회의록
회의 내용은 AI가 요약할 수 있다.

액션아이템
회의 결과 생성된 액션아이템은 수정 가능하다.
업무요청으로 전환할 수 있다.
액션아이템 상태는 회의록 내부 상태로 관리한다.
액션아이템 상태는 업무항목 상태와 별개이다.
액션아이템을 업무요청으로 전환해도 업무항목은 자동 생성되지 않는다.
업무요청 수락 후 사용자가 Weekly Tasks에 추가해야 업무항목이 된다.

회의 참여자 관리
다음 사용자가 참여자를 관리할 수 있다.
	•	회의 생성자
	•	Owner
	•	Manager

참여자 변경
	•	참여자 추가
	•	참여자 제거
가능

참여자 변경 알림
추가된 참여자와 제거된 참여자에게 알림을 발송한다.

10. 알림 정책
업무 요청
업무 요청 생성 시 업무요청 영역에 즉시 표시한다.

업무 수정
업무 참여자에게 알림을 발송한다.

업무항목 참여자 변경
기존 참여자와 신규 참여자에게 알림을 발송한다.

연차 신청
승인자에게 알림을 발송한다.

연차 승인
신청자에게 알림을 발송한다.

연차 반려
신청자에게 알림을 발송한다.

알림 보관
일반 알림 목록에는 최근 90일 알림만 표시한다.
데이터베이스에는 1년간 보관한다.
1년 초과 알림은 시스템 정책에 따라 아카이브 또는 삭제 대상이 된다.
사용자는 알림을 삭제할 수 없다.
