# WorkFlow Hub - 기획용 데이터 모델

## 목적

본 문서는 WorkFlow Hub의 핵심 엔티티와 관계를 기획 관점에서 정의한다.

구체적인 DB 타입, 인덱스, API 스펙은 개발 단계에서 별도 정의한다.

## 핵심 원칙

- 업무항목은 프로젝트 역할을 수행하지만 별도 프로젝트 엔티티는 없다.
- 업무항목에는 완료 상태가 없다.
- 작업세션만 완료 처리할 수 있다.
- 업무요청은 전달 팀으로 수신되고 Manager가 담당자를 배정한다.
- 업무요청 수락 후 업무항목은 자동 생성하지 않는다.
- 업무카테고리 변경은 예약 변경으로 저장하고 다음 월 1일에 반영한다.
- 작업세션은 물리 삭제한다.
- My Page에는 총/사용/잔여 연차 현황을 포함하지 않는다.

## 엔티티 목록

### Company

서비스를 사용하는 회사 또는 조직.

주요 필드:

- 회사 ID
- 회사명
- 생성일시

관계:

- 하나의 Company는 여러 Team을 가진다.
- 하나의 Company는 여러 User를 가진다.

### Team

회사 내 업무 조직 단위.

주요 필드:

- 팀 ID
- 회사 ID
- 팀명
- 활성 여부
- 생성일시

관계:

- 하나의 Team은 여러 User를 가진다.
- 업무요청은 전달 팀을 가진다.
- Manager는 소속 팀의 업무요청과 연차를 관리한다.

### User

서비스 사용자.

주요 필드:

- 사용자 ID
- 회사 ID
- 팀 ID
- 이름
- 이메일
- 권한: Owner, Manager, Member
- 입사일
- 활성 여부
- 생성일시

관계:

- User는 하나의 Team에 속한다.
- User는 업무요청의 요청자가 될 수 있다.
- User는 업무요청의 담당자가 될 수 있다.
- User는 업무항목의 참여자가 될 수 있다.
- User는 작업세션의 작성자가 될 수 있다.
- User는 회의 생성자 또는 회의 참여자가 될 수 있다.
- User는 연차 신청자가 될 수 있다.

### Work Category

작업세션 분석용 분류.

주요 필드:

- 카테고리 ID
- 회사 ID
- 카테고리명
- 상태: 사용중, 비활성화
- 적용 시작일
- 생성일시
- 최종 수정일시

관계:

- 작업세션은 하나의 Work Category를 가진다.
- 카테고리 변경은 Work Category Change로 예약 저장된다.

정책:

- Owner만 생성, 수정, 비활성화할 수 있다.
- 삭제는 불가능하다.
- 기존 작업세션의 카테고리 정보는 변경하지 않는다.

### Work Category Change

업무카테고리 예약 변경 정보.

주요 필드:

- 변경 ID
- 카테고리 ID
- 변경 유형: 생성, 수정, 비활성화
- 변경 내용
- 요청자 ID
- 요청일시
- 적용 예정일: 다음 월 1일
- 적용 상태: 대기, 적용 완료, 취소

관계:

- 하나의 Work Category Change는 하나의 Work Category 또는 신규 카테고리 생성 요청과 연결된다.

정책:

- 변경 요청은 즉시 사용 상태에 반영하지 않는다.
- 매월 1일에 적용 예정 변경을 실제 카테고리 상태에 반영한다.

### Work Request

팀 간 업무 전달 및 협업 요청.

주요 필드:

- 업무요청 ID
- 요청자 ID
- 요청 팀 ID
- 전달 팀 ID
- 담당자 ID
- 업무명
- 업무 상세내용
- 시작일
- 마감일
- 중요도: 일반, 높음, 긴급
- 상태: 미배정 업무, 수락 대기, 수락, 거절
- 거절 사유
- 기타 거절 상세 사유
- 생성일시
- 최종 수정일시

관계:

- Work Request는 하나의 전달 팀을 가진다.
- Work Request는 담당자 배정 전까지 담당자 ID가 비어 있을 수 있다.
- 담당자 배정 후 담당자가 수락 또는 거절한다.
- 수락된 Work Request는 사용자가 Weekly Tasks에 추가할 때 Work Item 생성의 기반이 된다.

정책:

- Owner와 Manager만 생성할 수 있다.
- 업무요청 수락 후 업무항목은 자동 생성하지 않는다.
- 업무요청 수정 시 기존 업무항목은 자동 변경하지 않는다.
- 거절 사유는 MVP에서 고정 목록을 사용한다.

### Work Request History

업무요청 변경 이력.

주요 필드:

- 이력 ID
- 업무요청 ID
- 변경자 ID
- 변경 유형: 생성, 담당자 배정, 수정, 수락, 거절
- 변경 전 값
- 변경 후 값
- 변경일시

관계:

- 하나의 Work Request는 여러 Work Request History를 가진다.

정책:

- 모든 업무요청 수정 이력은 저장한다.

### Work Item

프로젝트 역할을 수행하는 업무 단위.

주요 필드:

- 업무항목 ID
- 회사 ID
- 생성자 ID
- 원본 업무요청 ID
- 업무명
- 업무 설명
- 시작일
- 마감일
- 업무유형: 일반, 고정, 긴급
- 생성일시
- 최종 수정일시
- 삭제 여부

관계:

- Work Item은 여러 참여자를 가질 수 있다.
- Work Item은 여러 작업세션을 가질 수 있다.
- Work Item은 업무요청 기반으로 생성될 수 있다.

정책:

- 업무항목에는 완료 상태가 없다.
- 직접 생성 시 생성자를 참여자로 자동 등록한다.
- 업무요청 기반 생성 시 담당자를 참여자로 자동 등록한다.
- 현재 날짜가 업무 기간에 포함되면 진행중처럼 강조 표시한다.
- 마감일 경과와 미완료 작업세션 존재 조건을 만족하면 지연중으로 계산한다.

### Work Item Participant

업무항목 참여자 정보.

주요 필드:

- 참여 ID
- 업무항목 ID
- 사용자 ID
- 참여 시작일시
- 참여 종료일시
- 활성 여부

관계:

- 하나의 Work Item은 여러 Work Item Participant를 가진다.
- 하나의 User는 여러 Work Item에 참여할 수 있다.

정책:

- Owner와 Manager만 참여자를 추가하거나 제거할 수 있다.
- Member는 업무항목 참여자 변경 권한이 없다.

### Work Session

실제 작업 기록 단위.

주요 필드:

- 작업세션 ID
- 업무항목 ID
- 작성자 ID
- 업무카테고리 ID
- 세부업무명
- 시작시간
- 종료시간
- 소요시간
- 완료 여부
- 생성일시
- 최종 수정일시
- 삭제일시
- 삭제자 ID

관계:

- Work Session은 하나의 Work Item에 속한다.
- Work Session은 하나의 작성자를 가진다.
- Work Session은 하나의 Work Category를 가진다.

정책:

- 최초 생성 시 업무카테고리와 세부업무명만 입력한다.
- 시간은 더블클릭 후 수동 입력한다.
- 타이머 기능은 제공하지 않는다.
- 소요시간은 시작시간과 종료시간 기준으로 자동 계산한다.
- 작업세션만 완료 체크가 가능하다.
- 작업세션은 물리 삭제한다.

### Work Session History

작업세션 수정 이력.

주요 필드:

- 이력 ID
- 작업세션 ID
- 변경자 ID
- 변경 유형: 생성, 수정, 완료 변경, 삭제
- 변경 전 값
- 변경 후 값
- 변경일시

관계:

- 하나의 Work Session은 여러 Work Session History를 가진다.

정책:

- 사용자 화면에는 표시하지 않는다.
- 모든 수정 이력은 데이터베이스에 저장한다.

### Meeting

회의와 회의록의 기준 엔티티.

주요 필드:

- 회의 ID
- 회사 ID
- 팀 ID
- 생성자 ID
- 회의 제목
- 회의 유형
- 회의 설명
- 녹음 상태: 대기, 녹음중, 완료
- 회의 요약
- 핵심 논의사항
- 스크립트
- 생성일시
- 저장일시

관계:

- Meeting은 여러 회의 참여자를 가진다.
- Meeting은 여러 액션아이템을 가질 수 있다.

정책:

- Owner, Manager, Member 모두 회의를 생성할 수 있다.
- 회의 생성자, Owner, Manager가 회의 참여자를 관리할 수 있다.

### Meeting Participant

회의 참여자 정보.

주요 필드:

- 회의 참여 ID
- 회의 ID
- 사용자 ID
- 참여 상태
- 추가일시
- 제거일시

관계:

- 하나의 Meeting은 여러 Meeting Participant를 가진다.

정책:

- 참여자 추가/제거 시 대상자에게 알림을 발송한다.

### Meeting Action Item

회의 결과 도출된 실행 업무 후보.

주요 필드:

- 액션아이템 ID
- 회의 ID
- 업무명
- 업무 상세내용
- 담당자 ID
- 마감일
- 상태: 진행 예정, 진행 중, 완료
- 업무요청 전환 여부
- 전환된 업무요청 ID

관계:

- Meeting Action Item은 하나의 Meeting에 속한다.
- Meeting Action Item은 Work Request로 전환될 수 있다.

정책:

- 액션아이템은 업무항목이 아니다.
- 액션아이템 상태는 회의록 내부 상태이며 업무항목 상태와 별개이다.
- 업무요청으로 전환해도 업무항목은 자동 생성되지 않는다.

### Leave Request

연차 신청 및 처리 정보.

주요 필드:

- 연차 ID
- 신청자 ID
- 승인자 ID
- 연차 유형: 종일 연차, 오전 반차, 오후 반차
- 사용 날짜
- 신청 사유
- 상태: 승인 대기, 승인 완료, 반려, 취소
- 반려 사유
- 신청일시
- 승인/반려일시
- 취소일시

관계:

- Leave Request는 하나의 신청자를 가진다.
- Leave Request는 하나의 승인자를 가질 수 있다.

정책:

- 시간차는 MVP 범위에서 제외한다.
- Member 신청은 Manager가 승인한다.
- Manager 신청은 Owner가 승인한다.
- 연차 시작 전까지만 취소할 수 있다.
- Calendar에는 연차를 표시하지 않는다.
- My Page에는 총/사용/잔여 연차 현황을 표시하지 않는다.

### Notification

시스템 알림.

주요 필드:

- 알림 ID
- 수신자 ID
- 알림 유형
- 제목
- 내용
- 우선순위: 일반, 중요, 긴급
- 읽음 여부
- 발생일시
- 읽음일시
- 연결 대상 유형
- 연결 대상 ID

관계:

- Notification은 하나의 수신자를 가진다.
- Notification은 업무요청, 업무항목, 연차, 회의 등과 연결될 수 있다.

정책:

- 알림 클릭 시 읽음 처리한다.
- 사용자는 읽음 상태를 수동 변경할 수 있다.
- 전체 읽음 처리가 가능하다.
- 사용자는 알림을 삭제할 수 없다.
- 일반 알림 목록에는 최근 90일만 표시한다.
- DB에는 1년간 보관한다.

### Audit Log

권한 변경, 주요 데이터 변경, 운영상 중요한 행위를 기록하는 감사 로그.

주요 필드:

- 감사 로그 ID
- 수행자 ID
- 대상 유형
- 대상 ID
- 행위 유형
- 변경 전 값
- 변경 후 값
- 발생일시

관계:

- Audit Log는 여러 주요 엔티티 변경과 연결될 수 있다.

정책:

- 권한 변경 이력은 반드시 저장한다.
- 사용자 화면에 모든 감사 로그를 노출할 필요는 없다.

## 주요 관계 요약

```txt
Company
  -> Team
  -> User

Team
  -> Work Request 전달 팀
  -> User 소속 팀

Work Request
  -> User 요청자
  -> Team 전달 팀
  -> User 담당자
  -> Work Item 생성 기반

Work Item
  -> Work Item Participant
  -> Work Session

Work Session
  -> User 작성자
  -> Work Category

Meeting
  -> Meeting Participant
  -> Meeting Action Item
  -> Work Request 전환

Leave Request
  -> User 신청자
  -> User 승인자

Notification
  -> User 수신자
  -> 관련 엔티티
```
