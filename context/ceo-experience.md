# 대표(Owner) 화면 기획서

> WorkFlow Hub는 **하나의 앱**이지만 로그인한 사람의 역할(직원·팀장·대표)에 따라 화면이 달라진다.
> 이 문서는 그중 **대표(Owner) 화면**만 따로 정리한 기획서다.
> 대표 화면은 직원·팀장과 크게 갈라지므로, 구현 전에 "무엇이 왜 다른가 / 어떻게 생겼나"를 한곳에 못 박아 둔다.
>
> 작성일: 2026-06-19 · 갱신: 2026-06-20(홈 디벨롭 — 기간 전환·단계별 드릴다운·AI 추천 액션) · 기준 코드: `manager-react/`

---

## 1. 개요

### 1-1. 한 앱, 세 가지 얼굴

이 앱은 직원·팀장·대표를 **각각 다른 앱으로 만들지 않는다.** 페이지를 복제하지 않고,
"역할마다 다른 부분"만 한 파일(`manager-react/src/data/roles.js`)에서 분기한다.

- `ROLES.MEMBER`(직원) / `ROLES.MANAGER`(팀장) / `ROLES.OWNER`(대표)
- 화면 상단 역할 스위처로 전환하며 테스트한다.
- 직원은 "내 일"을, 대표는 "회사 전체"를 본다. 그래서 같은 메뉴 이름이라도 보이는 내용이 다르다.

### 1-2. 갈라짐의 3가지 유형

대표 화면이 직원·팀장과 달라지는 방식은 딱 3종류다. 이 문서 전체에서 이 분류를 쓴다.

| 유형 | 뜻 | 예 |
|---|---|---|
| **전용 컴포넌트** | 대표만 보는 완전히 다른 화면 | 홈, 리포트 |
| **조회범위차** | 화면은 같지만 대표는 "전사"를 본다 | 회의실(전체 회의록), (구)연차 |
| **권한차** | 화면은 같지만 대표가 더 많은 걸 할 수 있다 | 캘린더 일정 수정 권한 |

## 2. 대표 메뉴 한눈에 보기

대표 왼쪽 메뉴(사이드바)는 **6개**다.
> ⚠️ **연차관리·팀 현황 메뉴는 대표 화면에서 뺀다.** 연차는 **리포트 → 연차 탭**에서, 인력 현황은 **홈 핵심 지표·리포트 인력 탭**에서 본다.
> (직원·팀장 화면에는 두 메뉴가 그대로 있다. 대표 화면에서만 사라진다.)

| 메뉴 | 대표가 보는 것 | 직원·팀장과 비교 | 갈라짐 유형 |
|---|---|---|---|
| **홈** | 회사 전체 경영 대시보드 | 직원은 "내 업무" 홈 | 전용 컴포넌트 (`CeoDashboard`) |
| **프로세스** | 프로세스 템플릿 관리 (팀장과 동일) | 직원은 메뉴 없음 | 팀장과 동일 (`ProcessPage`) |
| **리포트** | 수익성·프로젝트·결재·인력·연차 보고서 (6탭) | 직원·팀장에겐 없는 메뉴 | **대표 전용 페이지** (`CeoReportCenter`) |
| **캘린더** | 일정(전체/팀/프로젝트 필터) | 화면 동일 | 권한차 (일정 수정 가능) |
| **회의실** | **모든 팀** 회의록 열람 | 직원·팀장은 소속 팀 회의만 | 조회범위차 |
| **마이페이지** | 대표 본인 정보·작업 히스토리 | 화면 동일 | 거의 같음 |

> 연차관리(`leave`)를 메뉴에서 빼면, `roles.js`의 대표 노출 페이지 목록에서 `'leave'`를 제외하고,
> 리포트 연차 탭에 승인 기능을 옮기면 된다. (구현 시 [5장](#5-2차미구현-항목) 참고)

## 3. 페이지별 기획

각 페이지를 ① **왜 다른가** ② **화면 구성** ③ **흐름·권한·이동** ④ **기획규칙 충돌 처리** 4블록으로 정리한다.

### 3-1. 홈 — 경영 대시보드 `CeoDashboard`

> 2026-06-20 디벨롭: 레이아웃 확정(3열 + 상단 KPI 스트립), **기간 전환(주간/월간/분기)**·
> **프로젝트 단계별 드릴다운**·**AI 추천 액션**을 새로 확정.
> **구현 상태(2026-06-20)**: 와이어프레임 v4 기준으로 `CeoDashboard` 재구현 완료 — KPI 스트립 5지표·기간 전환 탭·
> 3열(프로젝트 투자현황(예상→실제+오차율)·추이+결재함·AI 추천 액션)·결재 인라인 승인/반려/상세 구현됨.
> 아직 **2차**로 남은 것: ① 단계별 드릴다운(현재 행 클릭 시 기존 상세 슬라이드) ② 기간 전환의 실데이터 재집계·KPI 변화율(현재 추이는 예시·라벨 전환) ③ 진짜 예상 견적 기반 예상비용(현재는 정상 기간 투입을 예상으로 사용, 초과분=지연일) ④ AI 추천 버튼 실행 연계. → [5장](#5-2차미구현-항목).

**① 왜 다른가 — 대표가 홈에서 보고 싶은 것 (이 화면의 목적)**
직원·팀장 홈은 "오늘 내가 할 일"이다. 대표 홈은 회사를 위에서 내려다보는 경영 대시보드이며, 대표가 여기서 답을 얻고 싶은 질문은 3가지다.

- **질문 1. 직원들이 일 제대로 하나?** — 누가 놀고(가동률 낮음) 누가 과부하인가.
- **질문 2. 프로젝트가 잘 굴러가나?** — 진행/지연 상태, 어디서 막혔나.
- **질문 3. 이 프로젝트에 돈·인력이 얼마나 투입됐나?** — **몇 명 × 며칠 = 투자액**.

설계 원칙은 **"보는 대시보드"가 아니라 "결정하는 대시보드"** 다. 정보 뒤에 항상 "그래서 이렇게 하세요"가 붙고, **전체 파악 → 문제 발견 → 원인 분석 → 의사결정**이 화면 안에서 완결된다. 톤은 **감시 도구가 아니라 운영 최적화**(개인 저격 대신 운영 제안 — [④](#-기획규칙-충돌-처리)).

**② 화면 구성 — 헤더 + KPI 스트립 + 3열**
레이아웃을 확정한다(과거 TBD 해소). 상단 헤더(기간 네비) → 얇은 KPI 스트립(고정) → 그 아래 3열. **각 패널은 화면 높이에 꽉 차고, 넘치면 페이지가 아니라 패널 내부에서 스크롤**된다.

```
┌─ 헤더: 2026 6월 ‹ ›  [주간|월간|분기]        검색·알림·역할·프로필 ─┐
├─ KPI 스트립(고정·5칸): 가동률 78% │ 과부하2/여유3 │ 진행18/지연3 │ 오차+7% │ ▍총투입 9,249만 ─┤
├──────────────── 3열 (좌1.6 : 중1.1 : 우1.0) ─────────────────────┤
│ ┌─ 칸1 프로젝트 투자현황 ─┐ ┌─ 칸2 ───────────┐ ┌─ 칸3 AI추천 ─┐ │
│ │ [전체23][진행18][지연3] │ │ 업무 집중도 추이  │ │ ● 지연3건    │ │
│ │ (주)모아 ▓░ 7% +37%🔴 │ │ (가동률 막대,고정) │ │ '시안'정체   │ │
│ │  ▶ 클릭 → 단계별 드릴다운│ │ ───────────────  │ │ 💡인력 투입  │ │
│ │  ① AI진단 ② 예상vs실제 │ │ 결재함(스크롤)    │ │ [인력 배정]  │ │
│ │  ③ 범례  ④ 액션 버튼   │ │ [계약]그린푸드…   │ │ ● 여유 3명   │ │
│ └────────────────────────┘ └─────────────────┘ └─────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

| 칸 | 섹션 | 맡는 질문 |
|---|---|---|
| (상단) | **KPI 스트립** (5지표 고정) | 5초 스캔 |
| 1 (넓게) | **프로젝트 투자 현황** + 단계별 드릴다운 | 질문 2·3 |
| 2 | **업무 집중도 추이**(고정) + **결재함**(스크롤) | 질문 1 + 의사결정 |
| 3 | **AI 추천 액션** | 결정·실행 |

> 칸 비율 좌 1.6 : 중 1.1 : 우 1.0 — 대표가 가장 자주 보는 프로젝트가 가장 넓다. 결재함은 대기 건 있을 때만 보므로 좌(상시)·우(상시) 사이 가운데에 둔다.

**기간 전환 (주간 / 월간 / 분기)** — 헤더 탭으로 **데이터 범위·숫자 단위·추이 그래프 단위·AI 제안 성격**이 함께 바뀐다.

| 보기 | 프로젝트 범위 | 숫자 단위 | 추이 그래프 | AI 제안 |
|---|---|---|---|---|
| 주간 | 이번 주 작업분 | 주(수백만) | 일별(월~금) | 단기·전술("금주 인력 보강") |
| **월간**(기본) | 이번 달 전체 | 월 누적(천만) | 월별(최근 5개월) | 월 단위 운영 판단 |
| 분기 | 분기 누적 | 분기 누적(억) | 분기별(최근 4분기) | 전략("견적 재검토") |

기간이 길수록 → 숫자 단위 커지고(만→억) → 추이 단위 넓어지고(일→월→분기) → 제안이 전술→전략으로. ("매월 초 운영 전략 회의" 시나리오엔 주로 월간·분기.)

**KPI 스트립 (상단 고정 5지표)**

들어오자마자 조직 전체를 5초에 스캔하도록 얇게. 변화율은 색으로(초록=개선/빨강=악화).

| # | 지표 | 예시 | 보조 |
|---|---|---|---|
| 1 | 평균 가동률 | 78% | 전월 ▲5% · 목표 85% |
| 2 | 과부하 / 여유 인원 | 2 / 3명 | 재배분 가능 |
| 3 | 진행 / 지연 | 18 / 3건 | 지연 3건 |
| 4 | **예상 대비 실제(비용 오차)** | +7% | 목표 ±10% |
| 5 | **이번 달 총 투입**(강조 카드) | 9,249만 | 308 사람·일 · 예산 77% |

- 4번은 PRD 핵심 지표("예상 비용 대비 실제 비용 오차율 10% 이내")를 KPI로 박은 것. 5번(검은 강조 카드)은 가장 중요한 "돈" 정보.
- 4·5번의 **예상 비용**은 신규 데이터(예상 인원·기간)가 필요 → [5장](#5-2차미구현-항목)·[6장](#6-데이터-의존성-구현-참고).

**칸 1. 프로젝트 투자 현황 (메인 — "몇 명 × 며칠 = 얼마 투자됐나")**

- **탭이 곧 요약**: `전체(기본) · 진행 중 · 지연 · 이번 주 납기`, 각 탭에 건수(전체 23·진행 18·지연 3·납기 5). → 별도 요약 카드 불필요. "시작 전"은 **전체**에 포함, "승인 대기"는 프로젝트가 아니므로 **칸 2 결재함**으로.
- **테이블 행(4칸)**:

  | 1열 프로젝트/진행률 | 2열 예상/실제 | 3열 인원 | 4열 상태 |
  |---|---|---|---|
  | (주)모아커머스 ▓░ 7% | 488 → 668만 · **+37%🔴** | 4명·32일 | [지연] +2일 |

  1열 = 프로젝트명 + 진행률 막대 + %(작업세션 기반, "완료" 토글 없음 — 규칙). 2열 = **예상 비용 → 실제 비용 + 오차율**(+빨강/−초록). 3열 = 인원 + 공수(사람·일). 4열 = 상태 배지 + 납기/초과일수.
- **상태·강조 규칙**: 진행 중=파랑 배지(기본) / 지연=빨강 배지+왼쪽 빨간 바(납기 초과+미완료 세션) / **예산 초과=주황 배지+왼쪽 주황 바(예상 투입 < 실제 투입)**. 경고성(지연·초과·마감임박)만 색 포인트, 나머지 회색.
- **금액 ON/OFF 토글**: OFF면 ₩ 사라지고 **공수(사람·일)만** 남는다(금액 노출이 부담스러울 때).
- **투자액 계산 = 직급별 단가**: `Σ(투입 인원 직급 단가 × 투입 일수)`. 개인 월급은 앱에 안 넣고 직급별 단가만 사용.
  - 예) 김대리(25만/일)×8일 + 사원 2명(18만/일)×8일 = 200만 + 288만 = **488만**.
  - 공수·실제 투자액은 **실데이터**. **예상 비용**은 신규 입력(예상 인원·기간)이 필요 → 2차.

**칸 1 드릴다운 — 프로젝트 단계별 분석 (대표 대시보드의 핵심)**

대표가 "왜 +37% 초과됐지?" 궁금해 프로젝트 행을 클릭하면, **프로세스 단계별로 시간이 어디서 샜는지** 분해해 보여준다. "초과됐다"에서 끝나지 않고 "왜"까지 답해 의사결정으로 잇는다.

- **인터랙션**: 행 클릭 → 바로 아래 아코디언으로 펼침(한 번에 하나만, ▶→▼).
- **① AI 한 줄 진단(콜아웃)**: 원인을 한 문장으로. 예) *"'시안 제작' 단계에 예상의 2.6배 투입. 시안 반려 4회 반복이 원인, 전체 초과분의 78%."* 정상 프로젝트는 초록 긍정 메시지.
- **② 단계별 막대(예상 vs 실제)**: 단계명 + 예상일/실제일 + 막대 + 오차율. 회색(예상) 위에 보라(정상)/빨강(초과)를 겹쳐, 초과 단계만 튀어나오게. 미시작 단계는 "대기".
- **③ 범례**: 예상(회색) / 실제 정상(보라) / 실제 초과(빨강).
- **④ 액션 버튼**(문제 유형별): 지연→[단계 인력 보강][담당자 확인][상세] · 예산초과→[추가 예산 결재 올리기][계약 범위 확인][상세] · 정상→[상세].
- 이로써 **"보다→궁금→클릭→원인 파악→결정"** 이 화면 안에서 완결된다. (단계별 시간·예상 시간 집계는 신규 → [6장](#6-데이터-의존성-구현-참고).)

**칸 2. 업무 집중도 추이(고정) + 결재함(스크롤)**

- **업무 집중도 추이(상단 고정)**: 가동률 추이 막대(기간 단위 따라 일/월/분기). 현재 기간 막대만 보라 강조 + 한 줄 요약("6월 가동률 78% — 최근 5개월 중 최고").
- **결재함(하단 스크롤)**: 대표 승인이 필요한 **결재 안건**을 모아 인라인 [승인]/[반려]. (엔티티는 [6-1](#6-1-결재-안건-approval-item--신규-엔티티))
  - 유형 4종: **계약 · 예산 · 프로젝트 착수 · 프로젝트 종료** 승인.
  - 단순 금액만이 아니라 **판단 근거**(과거 거래·수익률·예상 원가/수익·여유 인원·일정 충돌)를 함께 제시 → 대표가 근거 보고 바로 결정.
  - 흐름: 팀장이 안건 등록 → 결재함 '대기' → 대표 [승인]/[반려](+사유) → 상태 변경 + 요청자 알림.
  - **연차는 결재함에 없다** → 리포트 연차 탭 전담. 승인 후 자동 연계(프로젝트 자동 생성 등)는 2차.

---

**칸 3. AI 추천 액션**

단순 경고 나열이 아니라 **문제 + 원인 + 추천 행동 + 실행 버튼**을 한 카드에. 대표가 클릭 한 번으로 처리.

```
● 지연 3건 모두 '시안 제작' 단계에서 정체
  💡 시안 단계 리소스 부족 — 여유 인력 투입 검토
  [ 인력 배정하기 ]
```

- 구성: 색 점(심각도) + 문제 요약 / 💡 추천 행동(보라 박스) / 실행 버튼(전체 너비).
- 심각도 색: 빨강=긴급(지연 정체) · 주황=주의(예산 초과·납기 집중) · 파랑=정보·기회(여유 인력 활용).

**③ 흐름·권한·이동**
- 홈은 대표 전용 컴포넌트(`CeoDashboard`)로 `roles.js`에서 Owner일 때 교체 렌더.
- 프로젝트 행 클릭 → 단계별 드릴다운(인라인). 액션 버튼 → 캘린더/상세/결재 등록 등으로 이동.
- 결재는 대표만 [승인]/[반려]. 안건 등록은 팀장(대표 직접 등록도 가능).

**④ 기획규칙 충돌 처리**
- 업무항목엔 "완료" 상태가 없으므로(규칙) 프로젝트 상태를 **진행 중·지연·시작 전**으로만 표현하고, 진행률은 작업세션 기반으로 계산한다.
- 매출은 데이터 미연동이라 노출 시 **"예시"** 표시. 단, 홈의 핵심 금액은 매출이 아니라 **직급별 단가로 계산한 투자**(실데이터). **예상 비용·오차율**은 예상 데이터가 들어와야 진짜 값이 된다(그 전엔 예시/비활성).
- **감시 도구로 비치지 않게 톤 관리**(PRD 리스크): ❌"김○○ 가동률 30%"(개인 저격) → ✅"여유 인력 3명 — 재배분 가능"(운영 제안). AI Brief·추천 액션 모두 **운영 최적화** 관점으로 표현.
- **계약·예산·프로젝트 착수/종료 승인**은 과거 "2차"로 미뤘으나, **결재 안건 엔티티로 MVP 편입**(이 결정으로 충돌 #4 정정 — [4장](#4-기획규칙-충돌-처리-확정-4건) 참조).

### 3-2. 팀 현황 — 대표 메뉴에서 제거 (결정 변경 2026-06-20)

- 미배정 신규 요청 같은 세부가 대표에게 다시 필요해지면 홈/리포트에 보조로 검토(전용 페이지 부활은 보류).

### 3-4. 리포트 — 경영 보고서 `CeoReportCenter` (대표 전용)

**① 왜 다른가 / 성격**
직원·팀장에겐 아예 없는, **대표만 보는 메뉴**다. 회사 전체를 주제별 보고서로 묶어 본다.
성격은 **"대표 혼자 경영을 깊이 들여다보는 분석 보고서"** — 홈은 한눈에, 리포트는 자세히.

**② 화면 구성 — 탭 6개** (수익성·프로젝트·결재·인력·연차·Export)

- **수익성** *(기존 "매출" 탭을 재편)*
  - ⭐ **메인은 프로젝트별 투입 원가**(직급별 단가 × 투입 일수). 홈 Project Status와 같은 계산식(`projectCost`)이며 **실데이터 기반**.
  - ⭐ **거래처별 투입 원가 비중**(**실데이터**): 프로젝트명에서 거래처명을 묶어 거래처별 프로젝트 수·투입 원가를 집계. 매출이 없어도 *"어디에 가장 많이 투자했나"*를 보여준다.
  - **매출·예상 매출·수익률**은 데이터 미연동이라 **"예시"**로 작게 표기. 매출이 연동되면 `매출 − 원가 = 수익`까지 완성(원가는 이미 진짜 값).
- **프로젝트**: 지표 4종 — `진행 중 건수 · 지연 건수 · 지연률 · 평균 진행 기간` + **전체 프로젝트 투자 상세표**(행마다 프로젝트명·인원·공수(사람·일)·투입 원가·진행률·납기, 하단 합계). 진행률은 작업세션 기반 **실데이터**.
  - ⚠️ **완료율·재작업률은 만들지 않는다.** 기획 규칙상 업무항목에 "완료" 상태가 없고 재작업 데이터도 없다. 진행률·진행 중/지연으로만 표현한다. ([④](#-기획규칙-충돌-처리) 참고)
- **결재**: 계약·예산·프로젝트 착수/종료 **결재 안건의 전체 이력**(대기·승인·반려) + 통계(대기/승인/반려 건수·승인 금액 합계). 홈 결재함과 같은 데이터(`approvalItems`)를 여기선 전체 이력으로 조회.
- **인력**: 지표 4종 — `1인 평균 작업시간(주) · 평균 가동률 · 과부하 인원(가동률 100% 초과) · 전체 인원` + **팀별 가동률**(막대, 100% 초과 시 경고 색). 모두 **실데이터**.
  - ⚠️ **야근 비율 → 과부하 인원으로 교체.** 출퇴근/근무시간 데이터가 없어 야근 비율은 산출 불가. 홈 KPI와 같은 가동률 기반 과부하 지표로 대체.
- **연차**: `오늘 연차 / 이번 주 연차 / 승인 대기 / 1인 연차 한도`.
  - ⭐ **연차는 여기서만 관리한다.** 연차관리 메뉴를 없앴고 **홈에도 연차를 두지 않으므로**, 연차 현황 보기와 **승인/반려**가 모두 이 탭에 모인다.
- **Export**: 리포트 내보내기(Weekly/Monthly/PDF/Excel/CSV) — *현재 "준비 중"*.

> **데이터 현실 구분(홈과 동일 원칙)** — 실데이터(진짜 값): 투입 원가·공수·프로젝트별 투자·거래처별 원가·진행률·가동률·과부하·연차·결재 금액. / 예시(미연동): 매출·예상 매출·수익률.

**③ 흐름·권한·이동**
- 리포트는 대표 전용 페이지(`report-center`)로 `roles.js`에서 Owner에게만 노출.
- 연차 승인 권한은 대표·팀장에게 있다(`canApproveLeave`). 대표는 팀장이 올린 연차까지 승인.

**④ 기획규칙 충돌 처리**
- 연차의 잔여/사용/한도 표시는 **My Page 금지 규칙과 무관**하다. 이건 개인 페이지가 아니라 **회사 전체 리포트**라서 허용.
- **완료율·재작업률·야근 비율은 데이터·규칙상 제외**하고 실데이터 지표(진행률·진행 중/지연·과부하 인원)로 대체한다. (완료 상태 없음 규칙 — `CLAUDE.md`)

### 3-5. 캘린더 (공유 화면 · 권한차)

**① 왜 다른가** — 대표는 전사를 보므로 화면이 **3단 구조**로 갈라진다(직원·팀장은 프로젝트별 타임라인). 일정을 **수정/삭제할 권한**도 더 있다(`canEditCalendar`).
**② 화면 구성** — **[팀 사이드바] → [선택 팀 프로젝트 리스트] → [월 캘린더]** 3단. 팀(전체/팀별+건수)을 누르면 가운데 리스트와 오른쪽 캘린더가 그 팀으로 필터되고, 프로젝트 리스트 항목을 클릭하면 우측에 **업무 상세 패널**(홈·일정 클릭과 동일 `DetailPanel`)이 열린다. 캘린더는 팀 색상 멀티데이 바. 회의·연차·반복 업무는 제외(규칙).
**③ 권한** — 현재는 조회 전용 UI라 권한차가 화면에 드러나지 않음. 편집 UI 추가 시 분기 적용([5장](#5-2차미구현-항목)).
**④ 기획규칙 충돌 처리** — **연차·업무요청·작업세션은 캘린더에 표시하지 않는다(규칙).** 납기·프로젝트 일정·필터는 표시 OK.

### 3-6. 회의실 (공유 화면 · 조회범위차)

**① 왜 다른가** — 화면은 같지만, 직원·팀장은 **소속 팀 회의만**, 대표는 **모든 팀 회의록을 전부** 본다(`canViewAllMeetings`).
**② 화면 구성** — 회의 목록·생성·상세(AI 요약·스크립트·액션아이템).
**③ 권한** — 회의 생성은 전 역할 가능. 참여자 추가/제거는 대표·팀장 가능(현재 UI 미구현, [5장](#5-2차미구현-항목)).
**④ 기획규칙 충돌 처리** — 회의 액션아이템 상태는 회의록 내부 상태일 뿐, 업무항목 상태와 별개(규칙).

### 3-7. 마이페이지 (공유 화면 · 거의 동일)

**① 왜 다른가** — 대표 본인 데이터일 뿐, 화면 구조는 직원과 같다.
**② 화면 구성** — 작업 히스토리, 생산성 분석, AI 업무 분석, 계정 정보.
**④ 기획규칙 충돌 처리** — **My Page에는 총/사용/잔여 연차 숫자를 넣지 않는다(규칙).** 히스토리 미니 캘린더엔 연차 "상태 표시"만 허용(숫자 X). 잔여 연차는 리포트에서 본다.

## 4. 기획규칙 충돌 처리 (확정 4건)

대표 화면 IA(최종 비전)와 기존 기획 규칙(`CLAUDE.md`)이 부딪히던 4가지를 2026-06-19 아래와 같이 확정했다.

1. **캘린더 연차 일정 → 제외 확정.** "캘린더엔 연차·업무요청·작업세션 표시 금지" 규칙 유지. 단 납기·프로젝트 일정, 전체/팀/프로젝트 필터는 위반 아니므로 OK.
2. **"완료 프로젝트 수/완료율" → 만들지 않음.** 업무항목엔 완료 상태 필드가 설계상 없고 별도 프로젝트 엔티티도 없음. → 대시보드는 **진행 중·지연·시작 전**으로만 표현. 업무항목에 완료 토글을 절대 만들지 않는다(규칙·데이터 구조 유지). 향후 경영 리포트 본격화 시 프로젝트 상위 엔티티 도입으로 확장 여지.
3. **My Page 미니 캘린더 연차 → 마크 "표시"만 OK, 잔여 연차 숫자 X.** 잔여 연차는 리포트에서 본다. (리포트의 "잔여 연차"는 개인 페이지가 아니라 회사 리포트라 규칙 무관.)
4. **(정정 2026-06-19) 계약·예산·프로젝트 착수/종료 승인 → MVP 편입.** 당초 "데이터 없음"을 이유로 2차로 미뤘으나, **결재 안건(Approval Item) 엔티티를 신규 추가**해 홈 Approval(대표 결재함)에서 다루기로 결정. ([6-1](#6-1-결재-안건-approval-item--신규-엔티티)) MVP 동작은 등록·승인/반려·기록·알림까지이며, 승인 시 자동 연계(프로젝트 생성 등)는 2차. 한편 연차는 홈 결재함이 아니라 **리포트 연차 탭 전담**으로 분리한다.

## 5. 2차(미구현) 항목

현재 코드에 "예시" 또는 "준비 중"으로 표시돼 있거나, 데이터 모델이 생겨야 할 수 있는 것들. 이번 구현엔 포함하지 않거나 별도로 다룬다.

- **연차관리 메뉴 삭제 + 리포트 연차 탭에 승인 기능 이전** ← 이번 변경의 핵심 구현 항목.
  - 대표 사이드바에서 `leave` 제거.
  - 리포트 연차 탭에 신청 목록·승인/반려 추가.
  - **홈에서 연차 완전 제거** — 홈 Approval은 결재 안건 전용. 연차 신청은 알림으로 인지.
- **프로젝트 투자 비용 계산(직급별 단가)** ← 홈 Project Status의 핵심.
  - ① `teamMembers`에 **직급 필드** 추가(현재 직무명 `role`만 있음).
  - ② **직급별 단가 표**(직급→1일 단가) 추가 + 대표 설정 화면.
  - ③ 홈 Project Status·KPI, **리포트 매출 탭**에 `공수(사람·일)`와 `≈ 금액(원가)` 병기 렌더링.
- **홈 Project Status 탭화** — `전체·진행 중·지연·이번 주 납기` 탭 + 탭별 건수. 기존 단일 리스트를 탭 필터로 교체(요약 카드 줄 제거).
- **결재 안건(Approval Item) 신규 추가** ← 홈 Approval(대표 결재함)의 핵심. (과거 "2차"였으나 MVP 편입)
  - 신규 데이터 `approvalItems`(유형 4종·필드·상태) — 정의는 [6-1](#6-1-결재-안건-approval-item--신규-엔티티).
  - 안건 등록(팀장) UI + 홈 결재함 인라인 [승인]/[반려](+사유) + 알림.
- **결재 안건 승인 시 자동 연계** — 착수 승인 시 프로젝트 자동 생성, 예산/계약 승인 시 비용 반영 등은 **2차**. (MVP는 상태 변경+기록+알림만)
- **(2026-06-20 디벨롭) 홈 기간 전환(주간/월간/분기)** — 탭에 따라 데이터 범위·숫자 단위·추이 단위·AI 제안 성격이 함께 바뀜([3-1 ②](#3-1-홈--경영-대시보드-ceodashboard)). 구현엔 **전기간 집계 로직**(전주/전월/전분기)과 **변화율 계산**(KPI ▲5% 등)이 필요.
- **(2026-06-20 디벨롭) 프로젝트 단계별 드릴다운** — 행 클릭 시 AI 진단 + 단계별 예상 vs 실제 막대 + 액션 버튼. 구현엔 **프로세스 단계별 시간 집계**와 **단계별 예상 시간/비용**(예상값)이 필요.
- **(2026-06-20 디벨롭) 예상 비용·오차율** — KPI 4번(예상 대비 실제)·칸1 2열·예산 초과 배지의 기준. **예상 인원·기간(예상 예산)** 데이터가 들어와야 산출 가능(그 전엔 비활성/예시).
- **(2026-06-20 디벨롭) KPI 스트립(5지표) + 업무 집중도 추이 + AI 추천 액션** — 현 구현(3칸 기본형)에서 KPI를 상단 스트립으로 재배치하고, 칸2를 추이+결재함으로, 칸3을 AI 추천 액션(문제+원인+행동+버튼)으로 재구성.
- **AI 추천 액션 실행 흐름** — "인력 배정하기" 등 버튼 클릭 후 실제 동작(팀장 지시 전달 등) 정의는 **2차**.
- **매출 데이터 연동** — 현재 매출·수익률는 전부 "예시" 값.
- **리포트 Export(PDF/Excel/CSV)** — 현재 버튼만 있고 "준비 중".
- **캘린더 일정 편집/삭제 UI** — 권한(`canEditCalendar`)은 있으나 UI 없음.
- **회의 참여자 추가/제거 UI** — 권한(`canManageParticipants`)은 있으나 UI 없음.

## 6. 데이터 의존성 (구현 참고)

대표 화면들이 공통으로 쓰는 데이터와 파생 규칙. (`manager-react/src/data/` 기준)

- **프로젝트 정의**: 반복(`recurringDays`) 아니고, 시작일(`start`) 있고, 회의 타입(`type==='회의'`) 아닌 **업무항목**을 "프로젝트"로 본다. (별도 프로젝트 엔티티 없음)
- **지연 판정**: 납기(`end`)가 오늘 이전인데 아직 끝나지 않은 작업세션(`!done`)이 남은 프로젝트.
- **진행률**: 프로세스가 있으면 "세션 찍힌 단계 수 / 전체 단계 수", 없으면 "완료 세션 / 전체 세션". (날짜가 아니라 실제 작업 기록 기준)
- **가동률**: 한 사람의 주간 업무항목 수 / 적정 용량(**4건 = 100%**).
- **투자액(실데이터)**: `Σ(직급 단가 × 투입 일수)`. **단계별 초과(드릴다운)**: 단계 실제 투입 시간 / 단계 예상 시간.
- **승인 대기**: 업무요청은 `status==='수락 대기'`, 연차는 `status==='승인 대기'`.
- 쓰는 데이터: `workItems`, `sessions`, `requests`, `assignmentRequests`, `leaves`, `teamMembers`, `processes`, **`approvalItems`(신규 — 아래)**.

**신규 필요 데이터 (2026-06-20 홈 디벨롭)** — 아래는 현재 없어 [5장](#5-2차미구현-항목)에서 단계적으로 마련한다.

| 항목 | 용도 |
|---|---|
| `teamMembers` 직급 필드 + 직급별 단가 표 | 투자액 계산 |
| 전기간 데이터(전주/전월/전분기) | 기간 전환·변화율(KPI ▲%) |
| 프로세스 단계별 시간 집계 | 칸1 드릴다운 |
| 단계별 예상 시간·비용 / 예상 인원·기간 | 예상 vs 실제·오차율·예산 초과 배지 |

> **데이터 현실 구분(리포트와 동일 원칙)** — 실데이터(진짜 값): 투입 원가·공수·진행률·가동률·과부하·결재 금액·단계별 실제 시간. / 예시·미연동: 매출·예상 매출·수익률·예상 비용(예상 데이터 입력 전까지).

### 6-1. 결재 안건 (Approval Item) — 신규 엔티티

홈 Approval(대표 결재함)의 데이터. 대표 승인이 필요한 의사결정 건을 담는다. **현재 코드엔 없으며 신규 추가 대상**(과거 "2차"였으나 MVP로 편입).

- **유형(`type`) 4종**: `계약 승인` · `예산 승인` · `프로젝트 착수 승인` · `프로젝트 종료 승인`.
- **공통 필드**: `id` · `title`(제목) · `type`(유형) · `requester`(요청자) · `requestedAt`(요청일) · `status`(`대기`/`승인`/`반려`) · `rejectReason`(반려 시 사유).
- **유형별 추가 필드**:
  - 계약 승인: `client`(거래처) · `amount`(계약 금액) · `period`(계약 기간)
  - 예산 승인: `amount`(금액) · `purpose`(용도) · `projectId?`(관련 프로젝트, 선택)
  - 프로젝트 착수 승인: `projectName` · `plannedHeadcount`(예상 인원) · `plannedPeriod`(예상 기간) · `plannedBudget`(예상 예산)
  - 프로젝트 종료 승인: `projectId`(대상 프로젝트) · `resultSummary`(결과 요약) · `actualInput`(실제 투입: 공수·금액)
- **흐름/권한**:
  - 등록: **팀장(Manager)이 대표에게 올림**(대표 직접 등록도 가능).
  - 결재: **대표(Owner)만** [승인]/[반려](+사유). 홈 결재함에 '대기' 안건만 표시.
- **승인 후 동작**: 상태 변경 + 기록 보관 + **요청자에게 알림**. 그 외 자동 연계(예: 착수 승인 시 프로젝트 자동 생성, 예산 승인 시 비용 반영)는 **2차**([5장](#5-2차미구현-항목)).
- **금액 표기**: 결재 안건의 금액은 입력값(실데이터). 매출 "예시"와 구분된다.
