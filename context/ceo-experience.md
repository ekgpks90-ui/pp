# 대표(Owner) 화면 기획서

> WorkFlow Hub는 **하나의 앱**이지만 로그인한 사람의 역할(직원·팀장·대표)에 따라 화면이 달라진다.
> 이 문서는 그중 **대표(Owner) 화면**만 따로 정리한 기획서다.
> 대표 화면은 직원·팀장과 크게 갈라지므로, 구현 전에 "무엇이 왜 다른가 / 어떻게 생겼나"를 한곳에 못 박아 둔다.
>
> 작성일: 2026-06-19 · 상태: 기획 확정(구현 대기) · 기준 코드: `manager-react/`

---

## 1. 개요

### 1-1. 한 앱, 세 가지 얼굴

이 앱은 직원·팀장·대표를 **각각 다른 앱으로 만들지 않는다.** 페이지를 복제하지 않고,
"역할마다 다른 부분"만 한 파일(`manager-react/src/data/roles.js`)에서 분기한다.

- `ROLES.MEMBER`(직원) / `ROLES.MANAGER`(팀장) / `ROLES.OWNER`(대표)
- 화면 상단 역할 스위처로 전환하며 테스트한다.
- 직원은 "내 일"을, 대표는 "회사 전체"를 본다. 그래서 같은 메뉴 이름이라도 보이는 내용이 다르다.

### 1-2. 갈라짐의 3가지 유형

대표 화면이 직원·팀장과 달라지는 방식은 딱 3종류다. 이 문서 전체에서 이 분류를 쓴다.

| 유형 | 뜻 | 예 |
|---|---|---|
| **전용 컴포넌트** | 대표만 보는 완전히 다른 화면 | 홈, 리포트 |
| **조회범위차** | 화면은 같지만 대표는 "전사"를 본다 | 회의실(전체 회의록), (구)연차 |
| **권한차** | 화면은 같지만 대표가 더 많은 걸 할 수 있다 | 캘린더 일정 수정 권한 |

## 2. 대표 메뉴 한눈에 보기

대표 왼쪽 메뉴(사이드바)는 **6개**다.
> ⚠️ **연차관리·팀 현황 메뉴는 대표 화면에서 뺀다.** 연차는 **리포트 → 연차 탭**에서, 인력 현황은 **홈 핵심 지표·리포트 인력 탭**에서 본다.
> (직원·팀장 화면에는 두 메뉴가 그대로 있다. 대표 화면에서만 사라진다.)

| 메뉴 | 대표가 보는 것 | 직원·팀장과 비교 | 갈라짐 유형 |
|---|---|---|---|
| **홈** | 회사 전체 경영 대시보드 | 직원은 "내 업무" 홈 | 전용 컴포넌트 (`CeoDashboard`) |
| **프로세스** | 프로세스 템플릿 관리 (팀장과 동일) | 직원은 메뉴 없음 | 팀장과 동일 (`ProcessPage`) |
| **리포트** | 매출·프로젝트·인력·연차 보고서 | 직원·팀장에겐 없는 메뉴 | **대표 전용 페이지** (`CeoReportCenter`) |
| **캘린더** | 일정(전체/팀/프로젝트 필터) | 화면 동일 | 권한차 (일정 수정 가능) |
| **회의실** | **모든 팀** 회의록 열람 | 직원·팀장은 소속 팀 회의만 | 조회범위차 |
| **마이페이지** | 대표 본인 정보·작업 히스토리 | 화면 동일 | 거의 같음 |

> 연차관리(`leave`)를 메뉴에서 빼면, `roles.js`의 대표 노출 페이지 목록에서 `'leave'`를 제외하고,
> 리포트 연차 탭에 승인 기능을 옮기면 된다. (구현 시 [5장](#5-2차미구현-항목) 참고)

## 3. 페이지별 기획

각 페이지를 ① **왜 다른가** ② **화면 구성** ③ **흐름·권한·이동** ④ **기획규칙 충돌 처리** 4블록으로 정리한다.

### 3-1. 홈 — 경영 대시보드 `CeoDashboard`

**① 왜 다른가 — 대표가 홈에서 보고 싶은 것 (이 화면의 목적)**
직원·팀장 홈은 "오늘 내가 할 일"이다. 대표 홈은 회사를 위에서 내려다보는 화면이며, 대표가 여기서 답을 얻고 싶은 질문은 딱 두 가지다.

- **목표 1. 직원들이 일 제대로 하나?** — 누가 놀고(가동률 낮음) 누가 과부하인가. ("농땡이 안 피우나")
- **목표 2. 프로젝트가 잘 굴러가나?** — 결국 **"이 프로젝트에 몇 명이 × 몇 개월 들어갔나 = 돈이 얼마나 투자됐나"**.

따라서 홈은 단순 "진행률 %"가 아니라 **사람(가동률)과 돈(프로젝트 투자)** 관점으로 구성한다. 특히 목표 2가 핵심이라 **Project Status를 "투자 현황"으로** 만든다.

**② 화면 구성 — 3개 칸(섹션)**
홈은 아래 3개 칸으로 구성한다. **매크로 레이아웃(3열 가로 vs 표 형태 등 칸의 배치/비율)은 아직 미확정(TBD)** 이며, 각 칸의 내부 설계만 아래에 확정한다.

```
┌─ Project Status ───────────┐ ┌─ KPI ──────────┐ ┌─ Approval(결재함) ─┐
│ [전체23][진행18][지연3][납기5]│ │ 가동률 78%      │ │ [계약] 그린푸드 납품 │
│ (주)그린푸드 배너   [진행 중] │ │ 과부하 2명      │ │  ₩3,000만   [승인][반려]│
│ ▓▓▓░░░░░░░ 13%             │ │ 여유 3명        │ │ [착수] 신규 앱 PJT   │
│ 담당 Jihye · 납기 06/17     │ │ 이번달 320사람일│ │  5명·3개월  [승인][반려]│
│ 👤5명 · 40사람·일   ≈488만  │ ├────────────────┤ │ [예산] 외주 디자인   │
│ ─────────────────────────  │ │ AI Brief        │ │  ₩500만     [승인][반려]│
│ 앱리뉴얼            [지연]   │ │ • 김○○ 가동30% │ │                   │
│ ▓░░░░░░░░░ 7%              │ │ • 박○○ 과부하  │ │                   │
│ 담당 Jihye · 납기 06/18     │ │ • 지연3건 점검   │ │                   │
│ 👤3명 · +2일 초과 488+180만 │ │ • ○○ 과투자 ⚠  │ │                   │
└─────────────────────────────┘ └────────────────┘ └───────────────────┘
   왼쪽: 프로젝트=투자(탭)         가운데: 사람 지표+경고   오른쪽: 대표 결재함
```

| 칸 | 섹션 | 맡는 목표 |
|---|---|---|
| 1 | **Project Status** (탭으로 분류) | 목표 2 — 프로젝트=투자 |
| 2 | **KPI** + **AI Brief** (위아래) | 목표 1 — 사람(가동률) |
| 3 | **Approval = 대표 결재함** | 대표 의사결정 |

---

**칸 1. Project Status (목표 2의 주인공 — "이 프로젝트에 얼마 투자됐나")**

- **탭으로 분류**: `전체(기본) · 진행 중 · 지연 · 이번 주 납기`. 각 탭에 **건수**를 붙여(전체 23 · 진행 18 · 지연 3 · 납기 5) **탭이 곧 요약 역할**을 한다. → 별도 요약 카드 줄 불필요.
  - "시작 전"은 별도 탭 없이 **전체**에 포함. "승인 대기"는 프로젝트가 아니므로 탭에서 빼고 **칸 3(결재함)**으로.
- **각 프로젝트 행에 진행률(막대 + %) 표시** — **모든 탭 공통**. (작업세션 기록 기반으로 계산. "완료" 토글은 만들지 않음 — 규칙 유지)
- **돈 표시**: **공수(사람·일)를 메인**으로, ₩ 금액은 **작게 보조**로 병기. **금액 ON/OFF 토글** 제공(평소 보조 표시, 부담스러우면 끄면 ₩ 사라지고 공수만 남음).
- **탭별로 보여줄 정보** (공통 정보를 강제로 넣지 않고 탭마다 필요한 것만):

  | 탭 | 각 행에 보여줄 것 | 상단 |
  |---|---|---|
  | 전체 | 진행률 · 상태 배지 · 인원 · 공수 · 투자(≈₩) · 납기 | 총 N개 · 총 투입 합계 |
  | 진행 중 | 진행률 · D-N(마감까지) · 인원 · **누적 / 예상 총 투입** | — |
  | 지연 | 진행률 · **+N일 초과** · **투입 + 추가 = 합계** · 담당자 | 지연 N건 · 추가비용 합계 |
  | 이번 주 납기 | 진행률 · D-N 마감 · 담당자 | — |

- **카드 가독성 원칙** (현재 UI의 깔끔함 유지):
  - 1줄: 프로젝트명 + 상태 배지 / 2줄: 진행률 막대 + % / 3줄: 회색 작은 글씨 메타(담당·납기·인원·공수·금액·D-N·초과일수 등).
  - **위 2줄(이름·진행률)은 건드리지 않는다.** 추가 정보는 **회색 메타 줄에만** 얹는다.
  - 경고성(지연 +N일, 추가비용, 마감 임박)만 **색**으로 포인트, 나머지는 회색.
- 클릭 시 캘린더/상세로 이동.

**투자(비용) 계산 = 직급별 단가**
- 계산식: `프로젝트 비용 = Σ(투입 인원 각자의 직급 단가 × 투입 일수)`. 개인 월급은 앱에 넣지 않는다(민감 정보 + 과한 설정).
  - 예) 디자인 개편 = 김대리(25만/일)×8일 + 사원 2명(18만/일)×8일 = 200만 + 288만 = **488만 원**
- **구현 시 준비물**(지금 데이터 없음 → [5장](#5-2차미구현-항목)): ① `teamMembers`에 **직급 필드** 추가 ② **직급별 단가 표** + 대표 설정 ③ 투입 일수는 작업세션/프로젝트 기간으로 산출.
- 공수·금액은 매출과 달리 **실데이터 기반**(예시 아님).

---

**칸 2. KPI + AI Brief (목표 1 — 사람)**

- **KPI(위)** — 사람(직원) 중심 4지표: `평균 가동률 · 과부하 인원 · 여유 인원 · 이번 달 총투입(공수·금액)`.
  - 가동률 기준: 주간 업무항목 4건 = 100%(현 코드 기준). 과부하 >100%, 여유 <50%.
- **AI Brief(아래)** — 직원+프로젝트 경고를 모두 다룸. 규칙 기반 자동 생성, 색 톤으로 심각도:
  - 농땡이(가동률 낮음) · 과부하 · 과투자/지연 프로젝트 · 납기 임박 · 여유 인원 분산 추천.

---

**칸 3. Approval = 대표 결재함**

- **연차는 홈에 두지 않는다.** 연차 승인/현황은 **리포트 연차 탭 전담**(신청이 오면 알림으로 인지). → 홈 결재함은 연차와 무관.
- **대표 승인이 필요한 "결재 안건"** 을 모아 인라인 [승인]/[반려]한다. (엔티티 정의는 [6장 결재 안건](#6-1-결재-안건-approval-item--신규-엔티티) 참조)
  - 유형 4종: **계약 승인 · 예산 승인 · 프로젝트 착수 승인 · 프로젝트 종료 승인**.
  - 표시: 유형 배지 + 제목 + 핵심값(금액/인원·기간 등) + [승인][반려](+사유).
- 흐름: 팀장이 안건 등록 → 대표 결재함에 '대기' 표시 → 대표 결재 → 상태 변경 + 요청자 알림.

**③ 기획규칙 충돌 처리**
- 업무항목엔 "완료" 상태가 없으므로(규칙) 프로젝트 상태를 **진행 중·지연·시작 전**으로만 표현하고, 진행률은 작업세션 기반으로 계산한다.
- 매출은 데이터 미연동이라 노출 시 **"예시"** 표시. 단, 홈의 핵심 금액은 매출이 아니라 **직급별 단가로 계산한 투자**이며 실데이터 기반이다.
- **계약·예산·프로젝트 착수/종료 승인**은 과거 "2차"로 미뤘으나, **결재 안건 엔티티로 MVP 편입**(이 결정으로 충돌 #4 정정 — [4장](#4-기획규칙-충돌-처리-확정-4건) 참조).

### 3-2. 팀 현황 — 대표 메뉴에서 제거 (결정 변경 2026-06-20)

당초 대표 전용 "전사 인력 현황"(`CeoCompanyStatus` — Team KPI·리소스·업무 편중·신규 요청)을 두었으나,
**대표 사이드바에서 팀 현황 메뉴를 제거**하기로 변경. → 대표 전용 `CeoCompanyStatus`는 **폐기(파일 삭제)**.

- 대표는 인력 현황을 **홈 핵심 지표(평균 가동률·과부하·여유 인원)**와 **리포트 인력 탭(1인 평균 작업시간·팀별 가동률)**에서 본다.
- 팀원 현황 상세 화면(`TeamStatusPage`)은 직원·팀장만 사용(그대로 유지).
- 미배정 신규 요청 같은 세부가 대표에게 다시 필요해지면 홈/리포트에 보조로 검토(전용 페이지 부활은 보류).

### 3-3. 프로세스 — 팀장과 동일 `ProcessPage`

**① 왜 같은가 (결정 변경 2026-06-20)**
당초 대표 프로세스는 병목 분석·AI 개선용 전용 화면(`CeoProcessPage`)으로 설계했으나,
**대표도 팀장과 똑같은 프로세스 템플릿 관리 화면(`ProcessPage`)을 쓰기로 변경**했다.
→ 대표 전용 `CeoProcessPage`는 **폐기(파일 삭제)**. 프로세스는 더 이상 대표가 갈라지는 화면이 아니다.

**② 화면 구성** — 팀장과 동일: 프로세스 템플릿 목록(단계 수) + 추가/수정/삭제.

**③ 권한** — 프로세스 템플릿 조회·관리는 Owner=O, Manager=O, Member=X(메뉴 자체 숨김). 대표·팀장 동일 동작.

> 병목 분석·AI 개선 같은 경영 분석이 다시 필요해지면 리포트 센터에 별도 탭으로 검토(전용 페이지 부활은 보류).

### 3-4. 리포트 — 경영 보고서 `CeoReportCenter` (대표 전용)

**① 왜 다른가**
직원·팀장에겐 아예 없는, **대표만 보는 메뉴**다. 회사 전체를 주제별 보고서로 묶어 본다.

**② 화면 구성 — 탭 5개**
- **매출**: 이번 달/예상 매출, 수익률 등 — *매출 수치는 현재 "예시" 값(데이터 미연동)*.
  - ⭐ **프로젝트별 투입 원가(직급별 단가 × 일수)를 함께 표시한다.** 홈 Project Status와 같은 계산식이며 **실데이터 기반**이다.
  - 매출이 연동되면 `매출 − 원가 = 수익`까지 완성. 지금은 **원가만 진짜 값**으로 보이고 수익률은 매출 연동 후. (상세 설계는 리포트 차례에 확정)
- **프로젝트**: 진행 중/지연/지연률/평균 진행 기간 + 지연 프로젝트 목록.
- **인력**: 1인 평균 작업시간(주), 평균 가동률, 전체 인원, 팀별 가동률.
- **연차**: 오늘 연차 / 이번 주 연차 / 승인 대기 / 1인 연차 한도.
  - ⭐ **연차는 여기서만 관리한다.** 연차관리 메뉴를 없앴고 **홈에도 연차를 두지 않으므로**, 연차 현황 보기와 **승인/반려**가 모두 이 탭에 모인다.
  - (현재는 숫자만 보여줌 → 승인 기능 추가 필요. [5장](#5-2차미구현-항목))
- **Export**: 리포트 내보내기(PDF/Excel 등) — *현재 "준비 중"*.

**③ 흐름·권한·이동**
- 리포트는 대표 전용 페이지(`report-center`)로 `roles.js`에서 Owner에게만 노출.
- 연차 승인 권한은 대표·팀장에게 있다(`canApproveLeave`). 대표는 팀장이 올린 연차까지 승인.

**④ 기획규칙 충돌 처리**
- 연차의 잔여/사용/한도 표시는 **My Page 금지 규칙과 무관**하다. 이건 개인 페이지가 아니라 **회사 전체 리포트**라서 허용.

---

### 3-5. 캘린더 (공유 화면 · 권한차)

**① 왜 다른가** — 화면은 직원과 똑같다. 대표는 일정을 **수정/삭제할 권한**이 더 있다(`canEditCalendar`).
**② 화면 구성** — 납기·프로젝트 일정, 전체/팀/프로젝트 필터.
**③ 권한** — 현재는 조회 전용 UI라 권한차가 화면에 드러나지 않음. 편집 UI 추가 시 분기 적용([5장](#5-2차미구현-항목)).
**④ 기획규칙 충돌 처리** — **연차·업무요청·작업세션은 캘린더에 표시하지 않는다(규칙).** 납기·프로젝트 일정·필터는 표시 OK.

---

### 3-6. 회의실 (공유 화면 · 조회범위차)

**① 왜 다른가** — 화면은 같지만, 직원·팀장은 **소속 팀 회의만**, 대표는 **모든 팀 회의록을 전부** 본다(`canViewAllMeetings`).
**② 화면 구성** — 회의 목록·생성·상세(AI 요약·스크립트·액션아이템).
**③ 권한** — 회의 생성은 전 역할 가능. 참여자 추가/제거는 대표·팀장 가능(현재 UI 미구현, [5장](#5-2차미구현-항목)).
**④ 기획규칙 충돌 처리** — 회의 액션아이템 상태는 회의록 내부 상태일 뿐, 업무항목 상태와 별개(규칙).

---

### 3-7. 마이페이지 (공유 화면 · 거의 동일)

**① 왜 다른가** — 대표 본인 데이터일 뿐, 화면 구조는 직원과 같다.
**② 화면 구성** — 작업 히스토리, 생산성 분석, AI 업무 분석, 계정 정보.
**④ 기획규칙 충돌 처리** — **My Page에는 총/사용/잔여 연차 숫자를 넣지 않는다(규칙).** 히스토리 미니 캘린더엔 연차 "상태 표시"만 허용(숫자 X). 잔여 연차는 리포트에서 본다.

## 4. 기획규칙 충돌 처리 (확정 4건)

대표 화면 IA(최종 비전)와 기존 기획 규칙(`CLAUDE.md`)이 부딪히던 4가지를 2026-06-19 아래와 같이 확정했다.

1. **캘린더 연차 일정 → 제외 확정.** "캘린더엔 연차·업무요청·작업세션 표시 금지" 규칙 유지. 단 납기·프로젝트 일정, 전체/팀/프로젝트 필터는 위반 아니므로 OK.
2. **"완료 프로젝트 수/완료율" → 만들지 않음.** 업무항목엔 완료 상태 필드가 설계상 없고 별도 프로젝트 엔티티도 없음. → 대시보드는 **진행 중·지연·시작 전**으로만 표현. 업무항목에 완료 토글을 절대 만들지 않는다(규칙·데이터 구조 유지). 향후 경영 리포트 본격화 시 프로젝트 상위 엔티티 도입으로 확장 여지.
3. **My Page 미니 캘린더 연차 → 마크 "표시"만 OK, 잔여 연차 숫자 X.** 잔여 연차는 리포트에서 본다. (리포트의 "잔여 연차"는 개인 페이지가 아니라 회사 리포트라 규칙 무관.)
4. **(정정 2026-06-19) 계약·예산·프로젝트 착수/종료 승인 → MVP 편입.** 당초 "데이터 없음"을 이유로 2차로 미뤘으나, **결재 안건(Approval Item) 엔티티를 신규 추가**해 홈 Approval(대표 결재함)에서 다루기로 결정. ([6-1](#6-1-결재-안건-approval-item--신규-엔티티)) MVP 동작은 등록·승인/반려·기록·알림까지이며, 승인 시 자동 연계(프로젝트 생성 등)는 2차. 한편 연차는 홈 결재함이 아니라 **리포트 연차 탭 전담**으로 분리한다.

## 5. 2차(미구현) 항목

현재 코드에 "예시" 또는 "준비 중"으로 표시돼 있거나, 데이터 모델이 생겨야 할 수 있는 것들. 이번 구현엔 포함하지 않거나 별도로 다룬다.

- **연차관리 메뉴 삭제 + 리포트 연차 탭에 승인 기능 이전** ← 이번 변경의 핵심 구현 항목.
  - 대표 사이드바에서 `leave` 제거.
  - 리포트 연차 탭에 신청 목록·승인/반려 추가.
  - **홈에서 연차 완전 제거** — 홈 Approval은 결재 안건 전용. 연차 신청은 알림으로 인지.
- **프로젝트 투자 비용 계산(직급별 단가)** ← 홈 Project Status의 핵심.
  - ① `teamMembers`에 **직급 필드** 추가(현재 직무명 `role`만 있음).
  - ② **직급별 단가 표**(직급→1일 단가) 추가 + 대표 설정 화면.
  - ③ 홈 Project Status·KPI, **리포트 매출 탭**에 `공수(사람·일)`와 `≈ 금액(원가)` 병기 렌더링.
- **홈 Project Status 탭화** — `전체·진행 중·지연·이번 주 납기` 탭 + 탭별 건수. 기존 단일 리스트를 탭 필터로 교체(요약 카드 줄 제거).
- **결재 안건(Approval Item) 신규 추가** ← 홈 Approval(대표 결재함)의 핵심. (과거 "2차"였으나 MVP 편입)
  - 신규 데이터 `approvalItems`(유형 4종·필드·상태) — 정의는 [6-1](#6-1-결재-안건-approval-item--신규-엔티티).
  - 안건 등록(팀장) UI + 홈 결재함 인라인 [승인]/[반려](+사유) + 알림.
- **결재 안건 승인 시 자동 연계** — 착수 승인 시 프로젝트 자동 생성, 예산/계약 승인 시 비용 반영 등은 **2차**. (MVP는 상태 변경+기록+알림만)
- **매출 데이터 연동** — 현재 매출·수익률는 전부 "예시" 값.
- **리포트 Export(PDF/Excel/CSV)** — 현재 버튼만 있고 "준비 중".
- **캘린더 일정 편집/삭제 UI** — 권한(`canEditCalendar`)은 있으나 UI 없음.
- **회의 참여자 추가/제거 UI** — 권한(`canManageParticipants`)은 있으나 UI 없음.

## 6. 데이터 의존성 (구현 참고)

대표 화면들이 공통으로 쓰는 데이터와 파생 규칙. (`manager-react/src/data/` 기준)

- **프로젝트 정의**: 반복(`recurringDays`) 아니고, 시작일(`start`) 있고, 회의 타입(`type==='회의'`) 아닌 **업무항목**을 "프로젝트"로 본다. (별도 프로젝트 엔티티 없음)
- **지연 판정**: 납기(`end`)가 오늘 이전인데 아직 끝나지 않은 작업세션(`!done`)이 남은 프로젝트.
- **진행률**: 프로세스가 있으면 "세션 찍힌 단계 수 / 전체 단계 수", 없으면 "완료 세션 / 전체 세션". (날짜가 아니라 실제 작업 기록 기준)
- **가동률**: 한 사람의 주간 업무항목 수 / 적정 용량(**4건 = 100%**).
- **승인 대기**: 업무요청은 `status==='수락 대기'`, 연차는 `status==='승인 대기'`.
- 쓰는 데이터: `workItems`, `sessions`, `requests`, `assignmentRequests`, `leaves`, `teamMembers`, `processes`, **`approvalItems`(신규 — 아래)**.

### 6-1. 결재 안건 (Approval Item) — 신규 엔티티

홈 Approval(대표 결재함)의 데이터. 대표 승인이 필요한 의사결정 건을 담는다. **현재 코드엔 없으며 신규 추가 대상**(과거 "2차"였으나 MVP로 편입).

- **유형(`type`) 4종**: `계약 승인` · `예산 승인` · `프로젝트 착수 승인` · `프로젝트 종료 승인`.
- **공통 필드**: `id` · `title`(제목) · `type`(유형) · `requester`(요청자) · `requestedAt`(요청일) · `status`(`대기`/`승인`/`반려`) · `rejectReason`(반려 시 사유).
- **유형별 추가 필드**:
  - 계약 승인: `client`(거래처) · `amount`(계약 금액) · `period`(계약 기간)
  - 예산 승인: `amount`(금액) · `purpose`(용도) · `projectId?`(관련 프로젝트, 선택)
  - 프로젝트 착수 승인: `projectName` · `plannedHeadcount`(예상 인원) · `plannedPeriod`(예상 기간) · `plannedBudget`(예상 예산)
  - 프로젝트 종료 승인: `projectId`(대상 프로젝트) · `resultSummary`(결과 요약) · `actualInput`(실제 투입: 공수·금액)
- **흐름/권한**:
  - 등록: **팀장(Manager)이 대표에게 올림**(대표 직접 등록도 가능).
  - 결재: **대표(Owner)만** [승인]/[반려](+사유). 홈 결재함에 '대기' 안건만 표시.
- **승인 후 동작**: 상태 변경 + 기록 보관 + **요청자에게 알림**. 그 외 자동 연계(예: 착수 승인 시 프로젝트 자동 생성, 예산 승인 시 비용 반영)는 **2차**([5장](#5-2차미구현-항목)).
- **금액 표기**: 결재 안건의 금액은 입력값(실데이터). 매출 "예시"와 구분된다.
