# WorkFlow Hub - 개발 인수인계 요약

## 1. 프로젝트 정의

### 서비스 목적

디자인 조직의 보이지 않는 업무를 데이터로 가시화하는 운영 분석 플랫폼.

### 핵심 문제

현재 디자인 조직은 결과물은 확인 가능하지만 다음 항목이 데이터로 관리되지 않는다.

- 어떤 업무를 수행했는지
- 얼마나 시간이 소요됐는지
- 어떤 업무에 리소스가 투입되는지

### 해결 방식

업무항목 -> 작업세션 -> 업무시간을 기록하여 조직 운영 데이터를 생성한다.

### 핵심 가치

WorkFlow Hub는 단순 업무 관리 툴이 아니라 업무 데이터를 수집하고 분석하는 SaaS이다.

## 2. 사용자 권한

### Owner

대표.

권한:

- 카테고리 관리
- 연차 승인
- 전체 데이터 조회
- 업무요청 생성
- 업무 수정
- 업무 삭제

### Manager

팀장.

권한:

- 업무요청 생성
- 업무 수정
- 업무항목 참여자 변경
- 팀 데이터 조회
- 연차 승인

### Member

직원.

권한:

- 본인 데이터 입력
- 본인 업무 수정
- 본인 작업세션 수정

## 3. 핵심 용어

### 업무항목 (Work Item)

업무 단위. Weekly Tasks에 생성된다.

예시:

- 쇼핑몰 리뉴얼
- 랜딩페이지 제작
- 고객사 수정 요청

### 작업세션 (Work Session)

실제 작업 기록 단위.

예시:

```txt
업무항목: 쇼핑몰 리뉴얼
작업세션: 메인 시안 제작 / 디자인 / 09:00 ~ 11:00
```

### 업무카테고리

분석용 분류. Owner만 관리한다.

예시:

- 기획
- 디자인
- 퍼블리싱
- 운영

- 오늘 남은 작업세션

### 업무요청 위젯

표시 항목:

- 업무명

- 중요도

기능:

- 수락
- 거절

### Weekly Tasks

주간 업무 계획 화면이며, 업무항목 관리 영역이다.

### Daily Todo

작업세션 관리 영역이다.

### 팀원 업무 현황

표시 항목:

- 오늘 작업시간
- 완료 작업세션
- 남은 작업세션
- 지연중
- 긴급업무

## 5. 업무항목 정책

### 생성 방식

직접 생성:

- 사용자가 생성한다.

업무요청 기반 생성:

- 업무요청 수락
- 원하는 날짜 선택
- Weekly Tasks 추가

### 자동 생성

업무요청 수락 후 업무항목은 자동 생성하지 않는다.

### 필수 입력값

- 업무명
- 시작일
- 마감일
- 업무유형

### 업무유형

- 일반: 기본값
- 고정
- 긴급

### 참여자

참여자는 직접 입력하지 않는다.

직접 생성:

- 생성자 자동 등록

업무요청 생성:

- 담당자 자동 등록

### 진행중

진행중 상태값은 없다. 현재 날짜의 카드만 강조 표시한다.

### 지연중

조건:

- 마감일 초과
- 미완료 작업세션 존재

표시:

- 지연중 태그

### 자동 숨김

조건:

- 마감일 초과
- 미완료 작업세션 없음

## 6. 작업세션 정책

### 생성 위치

Daily Todo.

### 필수 입력값

- 업무카테고리
- 세부업무명

### 최초 생성

시간 입력 없이 생성한다.

### 시간 입력

시간 영역을 더블클릭한 후 입력한다.

입력값:

- 시작시간
- 종료시간

### 시간 측정

타이머 기능은 없다. 전부 수동 입력한다.

### 소요시간

시작시간과 종료시간 기준으로 자동 계산한다.

### 완료 처리

작업세션만 완료 가능하다. 업무항목 완료 상태는 없다.

### 수정 이력

사용자 화면에는 표시하지 않고 DB에 저장한다.

### 삭제

가능 권한:

- Owner
- Manager
- 본인 작업세션 작성자

작업세션은 물리 삭제하지 않는다.

삭제 시 `deleted_at`, `deleted_by`를 기록하여 소프트 삭제한다.

소프트 삭제된 작업세션은 일반 사용자 화면에서 숨김 처리한다.

업무시간 및 생산성 분석에서는 기본 제외한다.

감사 및 이력 확인을 위해 DB에는 보관한다.

## 7. 업무요청 정책

### 생성 권한

- Owner
- Manager

### 필수 입력값

- 업무명
- 업무 상세내용
- 시작일
- 마감일
- 전달 팀
- 중요도

### 수신 및 배정

업무요청은 먼저 전달 팀으로 수신된다.

전달 팀의 Manager가 담당자를 배정한다.

담당자 배정 전에는 미배정 업무로 표시한다.

담당자 배정 후 담당자가 수락 또는 거절할 수 있다.

### 상태

- 미배정 업무
- 수락 대기
- 수락
- 거절

### 수락

업무는 자동 생성하지 않는다. 업무요청 위젯에 유지되며, 사용자가 직접 Weekly Tasks에 추가한다.

### 거절

거절 사유는 필수이다.

MVP에서는 고정 거절 사유 목록을 사용한다.

거절 사유:

- 일정 불가
- 담당 범위 아님
- 리소스 부족
- 요청 내용 불명확
- 기타

`기타` 선택 시 상세 사유 입력은 필수이다.

Owner가 거절 사유를 직접 관리하는 기능은 추후 확장으로 둔다.

### 수정

가능하다.

수정 항목:

- 업무명
- 상세내용
- 시작일
- 마감일
- 중요도

### 수정 반영

기존 업무항목은 자동 수정하지 않는다. 담당자가 직접 반영한다.

### 미응답

24시간:

- 1차 알림

48시간:

- 2차 알림
- 요청자 알림

자동 거절은 없다.

## 10. 회의실 정책

### 흐름

회의 생성 -> 녹음 -> AI 분석 -> 회의록 저장 -> 액션아이템 생성 -> 업무요청 전환.

### AI 생성 데이터

- 회의 요약
- 핵심 논의사항
- 액션아이템
- 스크립트

### 액션아이템

수정 가능하다.

액션아이템 상태는 회의록 내부 상태이다.

액션아이템 상태는 업무항목 상태와 별개이다.

액션아이템을 업무요청으로 전환해도 업무항목은 자동 생성되지 않는다.

업무요청 수락 후 사용자가 Weekly Tasks에 추가해야 업무항목이 된다.

업무항목에는 완료 상태가 없다.

### 업무요청 전환

필수값:

- 업무명
- 업무 상세내용
- 담당자
- 시작일
- 마감일
- 중요도

담당자 선택은 필수이다.

## 11. 업무카테고리 정책

### 관리 권한

Owner만 가능하다.

### 기능

- 생성
- 수정
- 비활성화

### 삭제

불가.

### 비활성화

신규 작업세션에서 선택할 수 없다.

### 적용 시점

카테고리 변경 사항은 매월 1일부터 적용한다.

생성, 수정, 비활성화 요청은 즉시 사용 상태에 반영하지 않고 예약 변경으로 저장한다.

예약 변경에는 `effective_from` 값을 저장한다.

`effective_from`은 다음 월 1일로 설정한다.

매월 1일에 예약 변경을 실제 카테고리 상태에 반영한다.

기존 작업세션 데이터의 카테고리 정보는 변경하지 않는다.

## 13. 알림 보관 정책

일반 알림 목록에는 최근 90일 알림만 표시한다.

데이터베이스에는 1년간 보관한다.

1년 초과 알림은 시스템 정책에 따라 아카이브 또는 삭제 대상이 된다.

사용자는 알림을 삭제할 수 없다.

## 15. 프로세스 관리 정책

### 개요

Manager가 공통 업무 프로세스를 템플릿으로 등록하고, 업무요청 생성 시 자동 매칭하여 단계별 배정을 지원하는 기능이다.

### 프로세스 템플릿

구성:

- 프로세스명
- 트리거 키워드 (1개 이상, 최대 10개)
- 프로세스 단계 (순서, 단계명, 담당 팀)

관리:

- 생성, 수정, 비활성화 가능
- 삭제는 불가
- 비활성화된 템플릿은 키워드 매칭 대상에서 제외

### 키워드 매칭

- 업무요청 업무명 입력 시 활성 템플릿 키워드와 자동 비교
- 매칭 시 프로세스 선택 UI 제안
- 선택은 필수가 아니며 무시 가능

### 업무요청 내 프로세스

- 하나의 업무요청에 하나의 프로세스 첨부
- 단계 상태: 대기 → 진행중 → 완료
- 단계는 순서대로 진행 (이전 단계 완료 후 다음 단계 진행중 전환)
- 각 단계별 담당자 별도 배정 가능
- 단계 완료 처리 권한: Manager, Owner

### 알림

- 단계 진행중 전환 시 → 해당 단계 담당자 알림
- 단계 완료 시 → Manager에게 다음 단계 시작 알림
